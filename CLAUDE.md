```bash
pio run              # build the project
pio run -t upload    # build and upload to ESP32
pio run -t monitor   # open serial monitor (115200 baud)
```

There are no tests — `test/` is empty.

## Architecture

This is a two-wheeled self-balancing legged robot running on an ESP32 (DOIT DevKit V1 / Arduino framework). The robot has two legs, each with two joints (hip + knee) driven by brushless motors, plus two wheel motors — 6 motors total, all communicating over CAN bus.

### Initialization sequence (`main.cpp`)

Modules are initialized in numbered priority order; higher-priority FreeRTOS tasks are created later to ensure data sources are ready before consumers:

1. **Serial** — debug printf at 115200 baud
2. **CAN** — TWAI (ESP32 CAN peripheral) at 1 Mbps; receives motor angle/speed frames
3. **IMU** — MPU6050 over I2C (pins 21/22), using DMP for Euler angles and gyro rates
4. **Motor** — 6 motor objects initialized with per-motor calibration (offset angles, torque ratios, back-EMF compensation curves, direction)
5. **Legs** — computes leg length/angle/speed from joint motor angles via MATLAB-generated forward kinematics
6. **Ctrl** — the main LQR + cascade PID balance controller
7. **BLE** — Bluetooth remote control (name: "Bot")

### Key data flow

```
CAN bus → Motor_Update() → motor angles/speeds
Motor angles/speeds → leg_position() / leg_speed() → LegPos (length, angle, dLength, dAngle)
MPU6050 DMP → imuData (pitch, roll, yaw + rates)
LegPos + imuData → stateVar → LQR feedback matrix (lqr_k) → motor torques via VMC (leg_vmc_conv)
BLE commands → target.speedCmd, target.yawSpeedCmd, target.legLength
```

### Task layout (FreeRTOS, 4ms base control period)

| Task | Priority | Period | Role |
|------|----------|--------|------|
| `IMU_Task` | 4 | 5ms | Poll MPU6050 DMP FIFO |
| `CAN_RecvTask` | 4 | 2ms | Poll TWAI for motor data frames |
| `Motor_SendTask` | 4 | 2ms | Compute back-EMF compensated voltages, send CAN torque commands (IDs 0x100 left, 0x200 right) |
| `LegPos_UpdateTask` | 2 | 4ms | Forward kinematics — motor angles → leg length/angle |
| `Ctrl_TargetUpdateTask` | 3 | 4ms | Ramp target speed/position from BLE commands |
| `CtrlBasic_Task` | 1 | 4ms | Main controller — LQR balance + cascade PID + VMC torque distribution |
| `Batteryvoltage_Task` | 5 | 200ms | Read battery voltage via ADS1115, adjust `motorOutRatio` |
| `Serial_Task` | 1 | 50ms | Debug serial output (commented-out printf lines) |

### Control structure

- **LQR**: `lqr_k(L0, K[12])` generates a 2×6 gain matrix from leg length `L0`. Applied to state vector `[theta, dTheta, x, dx, phi, dPhi]` (leg angle, leg angular velocity, wheel position, wheel speed, pitch, pitch rate). Outputs `[wheelTorque, hipTorque]`.
- **VMC** (Virtual Model Control): `leg_vmc_conv(F, Tp, phi1, phi4, T[2])` converts desired leg force `F` and hip torque `Tp` into joint torques `[T_knee, T_hip]`.
- **Cascade PID**: Outer loop on angle/position, inner loop on velocity. Used for leg length, leg angle difference, yaw, and roll.
- **Ground detection**: Estimates support force from leg thrust + IMU z-acceleration. Detects touchdown buffering to avoid bounce misdetection.

### MATLAB-generated code (`src/matlab_code/`, `include/matlab_code/`)

All files generated by MATLAB Coder 5.3 / Symbolic Math Toolbox. Do not hand-edit these — regenerate from MATLAB if the kinematics change.

- `leg_position(phi1, phi4, pos[2])` — forward kinematics: joint angles → leg length & angle
- `leg_speed(dphi1, dphi4, phi1, phi4, spd[2])` — Jacobian: joint velocities → leg extension/angular velocity
- `leg_vmc_conv(F, Tp, phi1, phi4, T[2])` — VMC force resolution: virtual leg force/hip torque → joint torques
- `lqr_k(L0, K[12])` — LQR gain matrix as cubic polynomial in leg length `L0`

## Calibration

The robot requires a one-time calibration after assembly. The documented procedure is in `DebugNote.docx`.

### Joint motor calibration (dir + offsetAngle)

**Joint reference position**: upper joint facing forward (car front), lower joint facing backward. The lower joints cannot physically face forward on an assembled robot.

**Process**:
1. Comment out `- motor->offsetAngle` in `Motor_Update()` ([motor.cpp:95](src/motor.cpp#L95))
2. Set tentative `dir` (start with the original values: left 1, right -1)
3. Position joints: upper forward, lower backward
4. Read raw motor angles from serial
5. Determine `dir` for each joint — rotate joint forward, if serial value increases → dir=1, decreases → dir=-1
6. Write dir back, re-burn, reposition, re-read serial values
7. Calculate offsetAngle per formula below
8. Fill offsetAngle, restore `- motor->offsetAngle`, verify

**offsetAngle formula** (all joints in reference position: upper forward, lower backward):
- **dir=1, forward**: `offsetAngle = reading`
- **dir=1, backward**: `offsetAngle = reading − π`
- **dir=−1, forward**: `offsetAngle = −reading`
- **dir=−1, backward**: `offsetAngle = π − reading`

**Verified values** (dir = 1, 1, -1, -1):
| Motor | offsetAngle | dir |
|-------|:---:|:---:|
| leftJoint[0] | 1.570 | 1 |
| leftJoint[1] | 1.061 | 1 |
| rightJoint[0] | 1.579 | -1 |
| rightJoint[1] | 11.576 | -1 |

**Verification**: after calibration, in reference position, upper joints = 0, lower joints = π. In balancing posture (body upright, legs vertical), theta ≈ 0 and `prot:0`.

### MPU calibration

1. Place IMU on a **level surface, completely still**
2. Ensure `mpu.CalibrateAccel(6)`, `mpu.CalibrateGyro(6)`, and `mpu.PrintActiveOffsets()` are uncommented in `IMU_Init()`
3. Burn, read serial, fill offset values into `mpu.setXAccelOffset()` etc.
4. Comment out the calibration calls for production

### Protection condition

`PROTECT_CONDITION` ([ctrl.cpp:298](src/ctrl.cpp#L298)) disables motors when `|theta| > π/4` (leg >45° from vertical) or `|pitch| > π/4` (body >45° tilted). The robot must be held upright at startup for the controller to engage. If `prot:1` appears in debug output, motors are blocked.

### Standup state machine

- `StandupState_Standup` → initial state, controller runs but protection logic may transition to `None` if legs are out of safe range
- `StandupState_None` → normal operation after standup completes
- BLE command `data[1]=88` triggers re-standup from `None` → `Standup`
- There is no separate standup task — the LQR controller handles it directly
